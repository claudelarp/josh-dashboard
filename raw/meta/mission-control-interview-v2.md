# Mission Control: Scoping Interview (v2)

Fill this in yourself, in this file. Not in a chat window. Several questions require you to go look something up or think for ten minutes, and a live chat pressures you into fast answers that are wrong.

Sections 1 and 2 are the raw material for everything the system will actually do. If you rush anything, do not rush those. Section 4 requires you to paste real artifacts, not describe them.

Where you don't know, write "don't know." The gaps are as informative as the answers.

---

## Section 0: Disqualifiers

If these answers are weak, the rest is theater.

**0.1** Name three specific moments in the last two weeks where you dropped something, forgot something, missed a window, or did worse work because information wasn't where you needed it. Dates and specifics. If you can't produce three, say so.

**0.2** What organizational system have you built or bought and then abandoned? For each: how long you used it, and the specific moment you stopped opening it. Not "I got busy." The actual mechanism of failure.

**0.3** What do you already open every single day without a reminder? This is your real substrate. Anything built elsewhere is fighting a habit you already have.

**0.4** Describe the last week where a drop deadline, a midterm, and a client deliverable collided. What actually got dropped? Who did you let down? What would a system have needed to know to prevent that, and be specific about whether the failure was informational or a straight capacity shortfall. If it was capacity, no system fixes it.

**0.5** You have eleven Claude skills, all serving the clothing brand and marketing work. Zero for school, fitness, or self-development. Why? Either (a) the brand is where the real work is and the other domains genuinely don't need this treatment, or (b) those domains are where you're struggling and you've avoided systematizing them. Which is it? These lead to opposite builds.

**0.6** If you didn't build this and instead spent those hours on the single highest-value thing available, what would that thing be? Write it down. Then answer whether building this is partly a way of not doing that.

## Section 1: Task Inventory

This is the most important section in the document. Every skill the system ever runs comes from here. Do not skip it, and do not do it from memory in one sitting.

For one week, at the end of each day, write what you actually did. Not what you meant to do. Then mark each item.

**1.1** Monday. What did you do, in order, in rough time blocks?

**1.2** Tuesday.

**1.3** Wednesday.

**1.4** Thursday.

**1.5** Friday.

**1.6** Saturday and Sunday, if different.

Now go back through all of it and mark each line:

- **R** if it repeats weekly or more often
- **H** if you hate doing it
- **I** if a competent intern could do it with written instructions
- **J** if it genuinely requires your judgment and cannot be delegated

**1.7** List everything marked R and H together. These are your first skills, in priority order.

**1.8** List everything marked I. For each, estimate what percentage of it could be handled by a system with the right context and connections. Not all or nothing. Thirty percent is a real answer.

**1.9** List everything marked J. Be skeptical of this list. For each item, state specifically what the judgment call is. If you can't articulate it, it probably belongs in I.

**1.10** What did you do more than three times this week? Even if it seemed trivial.

**1.11** What did you copy and paste this week, from where to where?

**1.12** What did you have to explain to someone, or to Claude, more than once?

**1.13** Take your single biggest recurring deliverable, whichever domain it lives in, and break it into every discrete step from start to finish. Ten to twenty steps. Then mark which steps are already covered by an existing skill, which could be, and which can't.

**1.14** If your order volume tripled next drop, what breaks first? If your client added a second location, what breaks first? If you took eighteen credits next semester, what breaks first?

## Section 2: Connection Map

The system can only act on what it can reach. Sketch this before answering, on paper or a whiteboard.

For each bucket below: what tool holds this, does it have an API or CLI, and do you have admin access to generate credentials?

**2.1 Revenue.** Shopify, payment processor, accounting, anything else. Which specific numbers do you check, how often, and which have ever changed a decision?

**2.2 Customers.** Where does customer data live? Email list, order history, DMs, support requests. How do customers currently reach you?

**2.3 Calendar.** One calendar or several? Class schedule, training schedule, client meetings, drop dates. Do they live in the same place?

**2.4 Communications.** Email (which provider), DMs on which platforms, texts, anything with your manufacturer, anything with the Blossom client. Where does each conversation type happen?

**2.5 Tasks and projects.** Do you have a task system at all right now, or is it notes and memory? If nothing, say nothing.

**2.6 Meetings and calls.** Client calls, group project calls, manufacturer calls. Are any recorded or transcribed? If not, would you?

**2.7 Knowledge.** Where does important information live that isn't in any of the above? Course materials, tech packs, brand guidelines, saved reference content, training programs, notes apps, camera roll, local files.

**2.8** Of everything listed above, rank the top three by how much you'd gain from the system being able to read them.

**2.9** Which of these would you be uncomfortable giving an automated system write access to? Name them explicitly.

**2.10** For the Blossom work specifically: are you operating inside their accounts, your own, or both? Whose credentials, and what have you agreed to in writing about access?

## Section 3: Data Provenance

For each metric below: where does the number live now, can it be exported or pulled programmatically, and if the honest answer is "I'd have to type it in," how many consecutive days will you actually do that?

**3.1** Lifts: sets, reps, weights. Current method?

**3.2** Running: mileage, pace, splits. Watch or app?

**3.3** Nutrition. Are you tracking now, or is this aspirational?

**3.4** Sleep. Wearable, phone, or estimate?

**3.5** Bodyweight and any other body metrics. Measurement frequency.

**3.6** Content performance, both accounts, plus Blossom's. Is this currently a manual screenshot habit?

**3.7** School: deadlines and grades. Is there an export, or is this transcribing a syllabus PDF once a semester? What's the grading composition per course this term?

**3.8** Reading, habits, religious practice, skill work. What would you actually record, in what form? "Habit tracker" is not an answer.

**3.9** Which of the above have you tracked consistently for more than sixty days at any point in your life? That's your realistic baseline. Everything else is a hypothesis.

## Section 4: Voice Capture

The system will draft in your name across contexts with different registers. It cannot infer this. Paste real artifacts, unedited, typos included.

**4.1** Two recent brand captions or posts, verbatim.

**4.2** Two recent personal-account captions or scripts, verbatim.

**4.3** One recent email or message to the Blossom client, verbatim.

**4.4** One recent message to your manufacturer or a supplier, verbatim.

**4.5** One piece of recent academic writing, a paragraph is enough.

**4.6** Anything you've written for yourself and no audience: journal entry, note to self, reflection. If you don't write for yourself, say so, because that answers a question about the self-development domain.

**4.7** For each of the above, name what makes it distinct. Where do the registers differ, and what would embarrass you if the system got wrong?

**4.8** Name three things you never say or write. Words, phrases, formatting habits you consider off-brand or not you.

## Section 5: Decision Rights and Permissions

**5.1** Decisions the system makes outright, no confirmation.

**5.2** Decisions where you want a recommendation with reasoning, but you decide.

**5.3** Areas where it should stay out of the way and only store information.

**5.4** Write access, file by file or system by system. What can be modified without asking? What must never be touched without review?

**5.5** For each external tool you connect: should the system use your credentials or its own scoped account with limited permissions? Which ones should be read-only? Answer this per tool, not in general.

**5.6** Cross-domain tradeoffs. If the system determines you should skip a lifting session to hit a client deadline, or push a drop a week because of a midterm, do you want it to say that? Or is that overreach you'd resent? Answer honestly, not aspirationally.

**5.7** What is the worst thing this system could do if it acted autonomously and got something wrong? Name the actual scenario. Then decide what permission model prevents it.

## Section 6: Cadence

**6.1** What Claude plan are you on? This determines your scheduled-run budget directly and is a hard constraint on what can run unattended.

**6.2** What would you want to be true when you wake up, that a system could have done overnight? Be specific. Not "be organized." Something like "a ranked list of the three things that matter today, with the reasoning visible."

**6.3** What should run weekly, without you asking?

**6.4** What should run per-cycle: per drop, per training block, per semester, per client reporting period?

**6.5** What should trigger on an event rather than a schedule? An order coming in, a class grade posting, a client message arriving, a sample shipping.

**6.6** Minimum scheduled interval is one hour and unattended runs are capped by plan. Given that budget, rank what gets the slots.

**6.7** Anything scheduled must succeed without you there to answer a question. Which of your answers to 6.2 through 6.5 actually meet that bar, and which would stall waiting for input?

## Section 7: Domain Scrutiny

### Clothing brand

**7.1** Drops per year, and the phase timeline from concept to shipped.

**7.2** Of the eleven existing skills, which have you invoked more than twice? Which did you build and never use again? This tells us whether you build tools you use or tools that feel productive to build.

**7.3** What breaks in a drop that no current skill covers?

**7.4** The single bottleneck right now: design, sampling, production, content volume, traffic, conversion, cash, or your own hours. Pick one. Inability to pick is itself a finding.

**7.5** Revenue and unit volume: last drop and trailing twelve months. Is the brand profitable, or funded by the contracting income?

**7.6** If the next drop fails completely, what happens? Financially and to the timeline.

### Blossom Medical

**7.7** Contractual scope: deliverables, cadence, fee, engagement length.

**7.8** What does the client actually receive from you? A report, a calendar, posts appearing, or only the work itself?

**7.9** The voiceover versus text-overlay question is still open from prior work. Resolved? If not, what's blocking it?

**7.10** Do you want this to scale to more aesthetics clients? Yes turns client work into a repeatable template and changes the architecture. No means don't build for it.

**7.11** How is footage consent currently tracked, and does that tracking need to live in this system? What are your obligations regarding patient-adjacent material?

### School

**7.12** Courses this term and what each is graded on.

**7.13** What is the GPA in service of? Recruiting, a specific program, self-respect? Changes how much the system should optimize for it.

**7.14** Recruiting timeline. You didn't mention it once, and for a McIntire student it's frequently the real deadline structure hiding behind coursework. Are you recruiting this cycle? For what, and what are the actual dates?

**7.15** How much coursework is group work with dependencies on other people?

### Fitness

**7.16** Aesthetic physique and running performance partially conflict. When they conflict, which loses? Answer now, in writing, so the system has a rule instead of relitigating weekly.

**7.17** Current bodyweight, main lifts, recent race times or weekly mileage. Targets and dates.

**7.18** Following a written program, or improvising? If written, whose? Do you want the system to program, or only to track?

**7.19** What has actually derailed training in the last twelve months?

### Self-development

**7.20** Rank religion, reading, hobbies, skill development, and habits by how much you actually care. Not aspirationally.

**7.21** For the top-ranked one only: what does progress look like, concretely, and what would you want to see on a screen?

**7.22** Now argue against including this domain at all. What's the case that these are things you do rather than things you manage, and that instrumenting them makes them worse?

## Section 8: Constraints

**8.1** Monthly budget for hosting, API, and subscriptions. A real number.

**8.2** Do you write code? What languages, what frameworks, and what happens when a cheap model hands you something broken at 11pm during midterms?

**8.3** Hours per week for maintenance, forever. Not the build sprint. Steady state.

**8.4** Phone-first or laptop-first? Most moments you'd check a mission control happen walking between classes with a phone. This one answer may determine whether a website is the right form factor at all.

**8.5** Who else is involved in any of this? Cofounder, designer, photographer, manufacturer contact, client-side contacts, training partner. For each: would they ever need to query this system, or is it yours alone?

**8.6** Where does this live and who can reach it? Grades, revenue, bodyweight, religious reflection, and a client's business data in one place. The client data has obligations attached.

**8.7** Have you ever set up a CLAUDE.md, a global skill, or an API integration before? What exists on your machine right now that this would build on?

---

## Section 9: Success, Instrumentation, and Kill Criteria

**9.1** Six months out, what is true that isn't true today? In numbers where possible.

**9.2** Three observable tests, adapted from people who've built these successfully. For each, state what "passing" looks like for you specifically:

- Could someone who needs an answer from you get a better answer by querying the system than by asking you?
- Have you stopped opening other tabs and apps to do work that could happen in one place?
- Has knowledge left your head, meaning you've stopped holding things in memory because you trust the system holds them?

**9.3** What instrumentation tells you whether 9.2 is being met? If you can't measure it, you'll rationalize.

**9.4** Kill criterion: a date and a condition. If by that date the condition is unmet, you shut it down. Write it now, before you're emotionally invested in the build.

**9.5** The smallest version that would be genuinely useful within one week. If you can't name one, the project is too abstract to start.

# Fable 5 Synthesis Prompt

Use after the interview above is filled in. Paste the completed interview and your current skill inventory together with this prompt in one message.

---

You are a systems architect who has watched most personal-productivity systems get abandoned inside sixty days. Your job on this pass is a build specification, not code and not encouragement. A cheaper model will write the implementation from your spec, so every architectural decision you skip becomes a decision that model makes badly.

## Inputs

1. A completed scoping interview covering five domains: a DTC clothing brand, marketing contracting for a medical aesthetics practice, university coursework, fitness and health, and personal development.
2. An inventory of eleven existing Claude skills, currently all serving the brand and marketing work.
3. Intent to connect this to an Obsidian vault operated by Claude Code.

## Architectural spine

Structure the entire specification around four layers, built strictly in this order. Later layers depend on earlier ones and cannot be specified before them.

**Context.** What the system knows: the person, the businesses, the voice registers, the current priorities, the money. Lives in markdown context files. Everything downstream fails without this.

**Connections.** What data it can reach. Derived from the interview's connection map. For each connection, specify the access method and justify it. Prefer direct API calls with a saved endpoint reference document over loading full MCP servers, because MCP loads every available function into context whether needed or not, and a markdown endpoint reference is dramatically cheaper to read. Specify credential handling: secrets in a gitignored environment file, never in chat, never committed. Where the interview flags sensitive access, specify a scoped service account with minimum necessary permissions and read-only where write isn't required, rather than the person's own full-permission credentials.

**Capabilities.** What it can produce. This means skills, and skills come from the interview's task inventory. Every skill you propose must map to a specific item marked repeated or hated in that inventory. If you cannot cite the inventory line, cut the skill.

**Cadence.** What runs unattended. Specified last, because nothing can be scheduled that isn't already a working capability.

## Skill specification requirements

For every skill, specify: folder location, whether project-scoped or global, the frontmatter name and description (the description is the trigger surface, so it must be specific enough to fire on the right request and not on adjacent ones), the step-by-step workflow, required reference files and their paths, and guardrail rules for known failure cases.

Respect progressive disclosure. The frontmatter is scanned on every request, the skill body loads only when the skill is selected, and reference files load only when the workflow needs them. Keep the skill body under 500 lines and push detail into references. Where a workflow repeatedly hits the same API endpoints or the same IDs, hardcode those into the skill rather than making it rediscover them each run.

Specify which skills are global rather than project-scoped, and justify each.

## Knowledge base specification

Specify the vault using the markdown-wiki pattern rather than a vector database: a raw folder for unprocessed sources, a wiki folder for synthesized linked notes, an index, and an operation log. Retrieval works by reading indexes and following links, not by similarity search. Specify the CLAUDE.md that explains the structure, the folder taxonomy, whether a hot cache of recent context is warranted, and a periodic lint pass that finds inconsistencies, gaps, and missing links.

Specify the write-back contract explicitly: what each agent reads at session start, what it writes at session end, what it must never overwrite. Human-authored notes are never destroyed. Agent-generated documents carry date, source, and links.

## Required output, in this order

**1. The cut list.** Before anything else, name what should not be built. Go domain by domain and identify which genuinely need system support and which are better served by an off-the-shelf tool or by nothing. Name every requested feature you are declining and why. This section should be uncomfortable to read. Treat any domain where the interview's own answers were vague or evasive as a candidate for cutting rather than for a design that accommodates the vagueness.

**2. The one-week version.** A v0 delivering real value within seven days, using only data sources the interview confirms already exist. If it requires building three things, it's too big.

**3. Context layer.** The context files, their schemas, and their update cadence.

**4. Connections layer.** Per connection: access method, credential handling, permission scope, and the reference document to be generated.

**5. Capabilities layer.** Full skill inventory per the specification requirements above. Explicitly state which of the eleven existing skills should be absorbed, refactored, or retired. Do not propose a new skill where an existing one can be extended.

**6. Cadence layer.** What runs scheduled versus event-triggered, at what interval, and within the plan's unattended-run budget as stated in the interview. For each scheduled item, confirm it can complete without human input, since anything that stalls waiting for an answer is not automatable. Distinguish items needing durable long-lived schedules from short-horizon monitoring loops. For anything running unattended against external systems, specify network access scope and justify anything beyond the minimum.

Separately: for each proposed automation, evaluate whether it needs an agent at all. A deterministic script beats an agentic workflow most of the time, and is cheaper and more reliable. State which items should be plain scripts.

**7. Interface decision.** Argue for or against a web application given the interview's answers on device usage, build capability, and maintenance hours. Consider three options: a persistent web app, a generated dashboard refreshed on a schedule, and a view rendered on demand by conversation. Pick one and defend it against the other two. Do not default to a web app because the request mentioned a website. If a dashboard is warranted, specify a disposable proof-of-concept first with a decision point: if the person doesn't open it repeatedly within two weeks, the custom build is cancelled.

**8. Build phases.** Sequenced, each producing something usable alone. Name dependencies. Estimate hours honestly for the stated skill level, then add fifty percent. Flag explicitly that productivity drops before it rises and name where in the sequence the dip lands, so it isn't mistaken for failure.

**9. Implementation prompts.** For each phase, a prompt written for a cheaper model. Each must be self-contained, including file structures, schemas, and conventions, because that model will not infer them. Specify acceptance criteria per phase so success is checkable without judgment.

**10. Self-improvement loop.** Specify the mechanism by which the system gets better through use. When a skill fails, the failure must be captured and written back into the skill so it cannot recur. Specify an audit routine that scores the four layers and identifies the weakest, and a routine that surfaces new automation candidates by asking what was repeated, what was manual, and what would break first under growth.

**11. Failure analysis.** The three most likely paths to abandonment, and a specific design decision mitigating each. Restate the kill criterion from the interview and specify the instrumentation needed to know whether it has been met.

## Tone

Do not validate the ambition. Assume the person has already decided to build this and needs the version of the plan that survives a semester, a product drop, and a training block landing in the same week. Where the interview answers are vague, evasive, or internally contradictory, say so and specify what would resolve the contradiction rather than designing around both possibilities. Where a proposed component exists because it sounds impressive rather than because a specific interview answer demands it, cut it and say why.

Output specification only. No code in this pass.
